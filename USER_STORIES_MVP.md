## 1. 数字底座MVP用户故事

### 1.1 设备接入

#### US-D-001: BLE蓝牙设备发现与连接

**优先级**: P0  
**PRD章节**: 2.1

**用户故事**:
> 作为系统，我需要能够发现并连接BLE蓝牙健康设备，以便采集用户健康数据。

**验收标准**:
- Given 用户启动了设备配对流程  
- When 系统扫描BLE设备时  
- Then 在10秒内返回范围内可发现设备列表，包含设备名称和信号强度  
- And 连接成功后，设备状态显示为"已连接"  
- And 设备断开连接时，系统在5秒内检测到并更新状态

- Given 一个已配对的BLE设备在范围内  
- When 用户请求重新连接时  
- Then 系统在3秒内完成连接  
- And 返回设备当前电池电量（如果支持）

#### US-D-002: WiFi设备接入

**优先级**: P0  
**PRD章节**: 2.1

**用户故事**:
> 作为系统，我需要能够发现并接入局域网内的WiFi健康设备，以便采集用户健康数据。

**验收标准**:
- Given 用户提供了WiFi设备的IP地址范围  
- When 系统执行局域网设备扫描时  
- Then 在15秒内返回在线设备列表，包含设备ID、类型和协议版本  
- And 连接成功后建立持久TCP连接  
- And 连接超时设置30秒，超时后自动重连最多3次

- Given WiFi设备响应异常  
- When 设备无响应超过30秒时  
- Then 系统记录错误日志并标记设备为"离线"  
- And 触发告警通知（如果配置）

#### US-D-003: 设备抽象层 - 统一数据模型

**优先级**: P0  
**PRD章节**: 2.1, 2.2

**用户故事**:
> 作为开发者，我需要统一的设备数据模型，以便简化数据处理逻辑。

**验收标准**:
- Given 不同协议设备上报的原始数据  
- When 数据经过抽象层处理时  
- Then 所有数据转换为标准DeviceData格式  
- And deviceId、deviceType、timestamp字段必填  
- And data字段包含设备特定数据，格式与设备类型匹配

- Given 设备类型"power_bike"上报数据  
- When 抽象层处理时  
- Then 输出包含 { cadence: number, power: number, resistance: number }  
- Given 设备类型"sleep_monitor"上报数据  
- Then 输出包含 { sleepDuration: number, deepSleepRatio: number, lightSleepRatio: number, awakeCount: number }

#### US-D-004: 设备注册与生命周期管理

**优先级**: P0  
**PRD章节**: 2.1

**用户故事**:
> 作为管理员，我需要管理设备的注册和注销，以便控制设备访问权限。

**验收标准**:
- Given 管理员发起设备注册请求  
- When 提交设备信息（名称、类型、协议、唯一标识）时  
- Then 系统验证设备唯一标识不重复  
- And 创建设备记录，初始状态为"待激活"  
- And 返回设备ID和API密钥

- Given 管理员发起设备注销  
- When 确认注销设备时  
- Then 系统软删除设备记录（状态变为"已注销"）  
- And 断开设备连接  
- And 保留设备历史数据

- Given 已注销的设备重新接入  
- When 设备使用相同唯一标识请求连接时  
- Then 系统可选择重新激活或拒绝

#### US-D-005: 设备状态监控

**优先级**: P1  
**PRD章节**: 2.1

**用户故事**:
> 作为管理员，我需要实时了解设备在线状态，以便及时处理故障。

**验收标准**:
- Given 设备处于连接状态  
- When 设备心跳超时60秒时  
- Then 系统更新设备状态为"离线"  
- And 记录离线时间和原因

- Given 设备状态变更  
- When 状态变化发生时  
- Then 管理后台实时更新设备状态指示器  
- And 推送状态变更通知（如果配置）

### 1.2 数据管道

#### US-D-006: 设备数据上报API

**优先级**: P0  
**PRD章节**: 2.3, 5.2

**用户故事**:
> 作为设备，我需要通过统一API上报数据，以便数据进入处理管道。

**验收标准**:
- Given 已注册设备上报数据  
- When POST /api/devices/{id}/data 请求发送时  
- Then 验证设备API密钥有效  
- And 数据进入Kafka队列  
- And 返回202 Accepted，包含数据接收时间戳

- Given 设备上报数据格式错误  
- When API接收请求时  
- Then 返回400 Bad Request  
- And 包含错误字段说明

- Given 设备上报数据量超过1MB  
- When API接收请求时  
- Then 返回413 Payload Too Large

#### US-D-007: 数据清洗与转换

**优先级**: P0  
**PRD章节**: 2.3

**用户故事**:
> 作为系统，我需要对原始设备数据进行清洗和转换，以便存储标准化的健康指标。

**验收标准**:
- Given 设备原始数据进入处理管道  
- When Flink任务处理数据时  
- Then 过滤无效数据（timestamp为null或未来时间）  
- And 转换数据格式为HealthMetrics  
- And 补全缺失字段（如时区转换）

- Given 数据清洗完成  
- When 数据写入TimescaleDB时  
- Then 数据延迟不超过5秒  
- And 数据持久化成功

#### US-D-008: 时序数据查询API

**优先级**: P0  
**PRD章节**: 2.3, 5.2

**用户故事**:
> 作为应用服务，我需要查询用户的健康指标时序数据，以便展示和分析。

**验收标准**:
- Given 应用服务请求用户健康数据  
- When GET /api/users/{id}/metrics?start=xxx&end=xxx 时  
- Then 验证用户授权有效  
- And 返回指定时间范围内的HealthMetrics列表  
- And 查询响应时间不超过2秒（30天数据量内）

- Given 请求包含聚合参数  
- When 指定aggregation=hourly|daily 时  
- Then 返回聚合后的统计数据（avg、min、max、count）

### 1.3 管理后台

#### US-D-009: 设备管理仪表盘

**优先级**: P0  
**PRD章节**: 5.1

**用户故事**:
> 作为管理员，我需要通过管理后台管理设备，以便监控整体设备状态。

**验收标准**:
- Given 管理员登录管理后台  
- When 查看设备管理页面时  
- Then 显示设备列表，包含：设备名称、类型、状态、最后在线时间、关联用户数  
- And 支持按状态筛选（全部/在线/离线/故障）  
- And 支持按设备类型筛选

- Given 管理员点击设备详情  
- When 查看单个设备详情时  
- Then 显示设备基本信息、连接历史、数据上报统计

#### US-D-010: 用户管理

**优先级**: P0  
**PRD章节**: 5.1

**用户故事**:
> 作为管理员，我需要管理用户账号，以便授权和监控用户健康数据。

**验收标准**:
- Given 管理员查看用户列表  
- When 分页加载用户列表时  
- Then 显示用户ID、姓名、注册时间、关联设备数、健康画像状态  
- And 支持搜索用户（按姓名、手机号）  
- And 支持查看用户详情（关联设备、健康档案、干预方案）

## 2. 干预引擎v0.1用户故事

### 2.1 个体健康画像

#### US-I-001: 创建用户健康画像

**优先级**: P0  
**PRD章节**: 3.1, 5.2

**用户故事**:
> 作为用户，我需要创建个人健康画像，以便获取个性化健康干预方案。

**验收标准**:
- Given 新用户首次使用系统  
- When 提交健康画像信息（年龄、性别、健康目标、生活习惯）时  
- Then 系统创建用户画像记录  
- And 返回画像ID和初始基线评估状态

- Given 用户提交画像信息  
- When 包含必填字段（age、gender）时  
- Then 保存成功并返回200  
- When 缺少必填字段时  
- Then 返回400并说明缺失字段

#### US-I-002: 用户画像数据更新

**优先级**: P0  
**PRD章节**: 3.1

**用户故事**:
> 作为用户，我需要更新我的健康画像，以便反映最新的健康状态。

**验收标准**:
- Given 用户更新健康画像  
- When 修改健康目标或偏好设置时  
- Then 系统更新画像并记录变更历史  
- And 变更后自动触发干预方案重新评估

### 2.2 基线评估

#### US-I-003: 基线数据采集

**优先级**: P0  
**PRD章节**: 3.2

**用户故事**:
> 作为系统，我需要采集用户连续7天的健康数据，以便建立健康基线。

**验收标准**:
- Given 用户完成画像创建  
- When 发起基线评估请求时  
- Then 系统创建基线任务，记录开始时间  
- And 每天自动汇总用户设备数据

- Given 基线采集进行中  
- When 用户数据不足1天时  
- Then 基线状态显示"采集中"，显示已采集天数  
- When 连续7天数据采集完成  
- Then 自动触发基线评估计算

#### US-I-004: 基线评估计算

**优先级**: P0  
**PRD章节**: 3.2

**用户故事**:
> 作为系统，我需要根据7天基线数据计算健康评分，以便生成干预方案。

**验收标准**:
- Given 7天基线数据采集完成  
- When 系统计算基线评估时  
- Then 输出心血管、代谢、睡眠、骨骼肌肉、压力5个维度评分(0-100)  
- And 计算综合健康评分(加权平均)  
- And 识别风险因素（如久坐、高压力）  
- And 生成风险预警（如有）

- Given 基线评估完成  
- When 用户查看基线报告时  
- Then 显示各维度评分、评分解释、改善建议  
- And 报告生成时间不超过10秒

### 2.3 运动干预

#### US-I-005: 运动处方生成

**优先级**: P0  
**PRD章节**: 3.3, 4.1

**用户故事**:
> 作为用户，我需要获取个性化的运动处方，以便改善健康状况。

**验收标准**:
- Given 用户完成基线评估且健康目标包含"运动"  
- When 用户请求运动处方时  
- Then 系统基于用户画像+基线评分生成运动方案  
- And 方案包含：运动类型、时长、强度、频率、心率目标  
- And 方案关联具体设备（如果用户已绑定运动设备）

- Given 用户健康目标为"减重"、基线代谢评分65  
- When 生成运动处方时  
- Then 推荐方案包含：每周4次有氧运动、每次30-45分钟、强度moderate、消耗300-500kcal  
- And 方案生成时间不超过5秒

#### US-I-006: 运动数据采集

**优先级**: P0  
**PRD章节**: 4.1

**用户故事**:
> 作为系统，我需要采集用户运动过程中的实时数据，以便评估运动效果。

**验收标准**:
- Given 用户开始执行运动处方  
- When 设备上报运动数据（心率、时长、消耗卡路里）时  
- Then 系统实时记录运动数据  
- And 计算实时运动强度（低/中/高）  
- And 记录运动完成状态

- Given 运动过程中心率超过目标上限  
- When 心率数据到达时  
- Then 系统记录警告事件  
- And 在运动报告中体现

#### US-I-007: 运动效果评估

**优先级**: P1  
**PRD章节**: 3.4, 4.1

**用户故事**>
> 作为用户，我需要了解我的运动效果，以便调整运动计划。

**验收标准**:
- Given 用户完成运动处方  
- When 查看运动效果报告时  
- Then 显示：运动依从性（实际完成次数/计划次数）、生理指标变化、效果评分  
- And 对比基线相关指标（如静息心率变化）

- Given 运动依从性>=80%  
- Then 效果报告显示"依从性良好"  
- Given 依从性<50%  
- Then 显示提醒，建议调整方案

### 2.4 睡眠干预

#### US-I-008: 睡眠数据采集

**优先级**: P0  
**PRD章节**: 4.2

**用户故事**:
> 作为系统，我需要采集用户的睡眠数据，以便评估睡眠质量。

**验收标准**:
- Given 用户佩戴睡眠监测设备入睡  
- When 设备上报睡眠数据时  
- Then 记录入睡时间、醒来时间、深睡时长、浅睡时长、清醒次数  
- And 第二天自动生成睡眠报告

- Given 睡眠数据不完整（如仅3小时数据）  
- Then 标记数据质量并提示用户

#### US-I-009: 睡眠质量评分

**优先级**: P0  
**PRD章节**: 4.2

**用户故事**:
> 作为用户，我需要了解我的睡眠质量评分，以便跟踪改善进展。

**验收标准**:
- Given 用户完成睡眠  
- When 系统计算睡眠质量时  
- Then 计算睡眠质量评分(0-100)  
- And 评分逻辑：深睡比例30%、睡眠时长25%、入睡时间规律性20%、醒来次数15%、其他10%  
- And 生成睡眠质量报告

- Given 评分>=80分  
- Then 显示"优秀"，绿色标识  
- Given 评分60-79分  
- Then 显示"一般"，黄色标识  
- Given 评分<60分  
- Then 显示"需改善"，红色标识

#### US-I-010: 睡眠处方生成

**优先级**: P0  
**PRD章节**: 3.3, 4.2

**用户故事**:
> 作为用户，我需要获取个性化的睡眠改善方案，以便提升睡眠质量。

**验收标准**:
- Given 用户完成基线评估且健康目标包含"睡眠改善"  
- When 用户请求睡眠方案时  
- Then 系统生成个性化睡眠方案  
- And 方案包含：建议入睡时间、建议起床时间、睡前仪式、环境建议

- Given 用户基线睡眠评分55、入睡时间23:30  
- When 生成睡眠处方时  
- Then 推荐：22:30入睡、06:30起床、21:00停止电子设备、21:30轻度拉伸  
- And 方案生成时间不超过5秒

### 2.5 效果评估闭环

#### US-I-011: 干预方案执行追踪

**优先级**: P0  
**PRD章节**: 3.4

**用户故事**:
> 作为系统，我需要追踪用户的干预方案执行情况，以便评估干预效果。

**验收标准**:
- Given 用户获得干预方案  
- When 用户执行方案时  
- Then 系统记录每日执行情况（完成/部分完成/未完成）  
- And 记录执行数据（运动数据、睡眠数据）

- Given 连续3天未执行方案  
- Then 系统发送提醒通知  
- And 记录依从性下降事件

#### US-I-012: 方案动态调整

**优先级**: P1  
**PRD章节**: 3.4

**用户故事**>
> 作为系统，我需要根据用户执行效果自动调整干预方案，以便提升干预有效性。

**验收标准**:
- Given 用户执行方案满2周  
- When 系统评估干预效果时  
- Then 对比基线指标与当前指标  
- And 根据规则引擎判断是否需要调整方案  
- And 生成调整建议（如降低运动强度、调整睡眠时间）

- Given 运动干预后静息心率上升超过10%  
- Then 系统建议降低运动强度20%  
- Given 睡眠干预后睡眠质量评分无改善  
- Then 系统建议调整睡前仪式或推荐就医

#### US-I-013: 健康报告生成

**优先级**: P1  
**PRD章节**: 3.4, 5.2

**用户故事**:
> 作为用户，我需要定期查看健康报告，以便了解整体健康状况。

**验收标准**:
- Given 用户请求健康报告  
- When GET /api/users/{id}/reports/weekly 时  
- Then 生成周报，包含：本周运动统计、睡眠统计、健康指标变化、方案执行情况  
- And 报告生成时间不超过15秒

- Given 用户请求月度报告  
- When GET /api/users/{id}/reports/monthly 时  
- Then 生成月报，包含：月度和趋势、干预效果评估、下月建议  
- And 报告包含可视化图表数据

## 4. MVP边界说明

### 4.1 数字底座MVP边界

**不支持**:
- Zigbee协议设备
- OTA固件升级
- 跨区域容灾部署
- 超过30天的历史数据查询

**支持**:
- BLE蓝牙和WiFi协议设备
- 5款核心设备（功率车、普拉提床、小米手环、睡眠监测垫、智能体脂秤）
- 单节点Kafka和TimescaleDB部署

### 4.2 干预引擎v0.1边界

**不支持**:
- 复杂ML模型（仅简单协同过滤）
- 除运动和睡眠外的其他干预场景
- 实时个性化方案推荐（基于规则的简单方案）
- 医疗诊断建议

**支持**:
- 规则引擎驱动的方案生成
- 7天基线评估
- 周/月健康报告
- 效果评估与简单方案调整
