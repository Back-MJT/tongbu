## 2. Phase 3 入口条件（修订）

### CEO 反馈核心问题：原前提条件过于乐观

原版本 Phase 3 启动依赖于「首批试点客户至少 1 家签约」，但 BD 窗口（4/12-4/14）结果不确定。修订后的 Phase 3 **不再以签约为前提**，而是根据 BD 结果分为两条执行路径。

### BD 结果评估节点：4 月 15 日

| 场景 | 触发条件 | Phase 3 重心 |
|------|----------|--------------|
| **乐观场景** | 4/15 前有客户签署 Pilot MOU | 路径 A：Pilot 客户技术接入 + C端小程序同步开发 |
| **保守场景** | 4/15 前无客户签署 | 路径 B：C端小程序 MVP + 持续 BD + 研究深度 |

**原则**：两条路径共享 C端小程序 MVP 工作流，差异仅在是否分流工程资源到 Pilot 接入。

---

## 3. Phase 3 战略重心（修订）

> **聚焦 C端小程序 MVP，建立触达终端用户的能力。干预引擎 v2 方向锁定 GUIDE paradigm，从规则驱动转向生成式体验。**

### 修订原则（来自 CEO 反馈）

1. **Scope Reduction**：等保认证（Phase 4）、行业标准制定（Phase 4）移出 Phase 3；聚焦可验证的核心交付物
2. **优先级锁定**：① C端小程序 MVP → ② Pilot 接入（乐观场景）→ ③ 研究深度
3. **时间线具体化**：每周明确起止
4. **GUIDE Paper 整合**：Generative Experience paradigm 注入干预引擎 v2 研究

---

## 9. Phase 3 vs Phase 4 边界（明确）

以下工作项明确移至 Phase 4，不在 Phase 3 范围内：

- ❌ 信息安全三级等保认证 → Phase 4

- ❌ 开放平台 SDK 发布 → Phase 4
