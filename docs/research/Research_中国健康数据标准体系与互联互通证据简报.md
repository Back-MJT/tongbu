## 一、中国健康数据标准体系全景

### 1.1 体系架构（三层）

```
国家卫生健康标准委员会
        ↓
┌──────────────────────────────────────┐
│ 第一层：GB/T 国家标准（强制/推荐）        │
│ 第二层：WS/T 卫生行业标准（推荐）         │
│ 第三层：团体标准（企业/协会自发）          │
└──────────────────────────────────────┘
        ↓
国际对接：HL7 FHIR（全球医疗数据互操作标准）
```

### 1.2 与健身器械直接相关的GB/T标准

| 标准号 | 名称 | 适用场景 | 与健身器械的关系 |
|--------|------|----------|-----------------|
| GB/T 33000 | 企业数据安全能力成熟度模型 | 数据安全管理 | 数据合规框架 |
| GB/T 35273 | 信息安全技术 个人信息安全规范 | 隐私保护/PIPL合规 | 用户运动数据合规 |
| GB/T 33136 | 信息技术 学习、教育和培训 数据与坐标 | 教育/体育数据格式 | 可类比用于健身数据 |
| GB/T 4208 | 外壳防护等级（IP代码） | 硬件标准 | IMU模组防护等级 |
| GB/T 17626 | 电磁兼容 | 硬件EMC | BLE模组EMC合规 |

### 1.3 与健身/健康数据直接相关的WS/T标准

| 标准号 | 名称 | 适用场景 | 与健身器械的关系 |
|--------|------|----------|-----------------|
| WS/T 447 | 基于电子健康档案的区域卫生信息平台 | 医院-体检数据互通 | 健身数据→医疗数据互通接口 |
| WS/T 500 | 电子健康档案数据集元数据规范 | 数据集规范 | 运动数据集元数据设计参考 |
| WS/T 456 | 健走支持环境指南 | 健康促进 | 运动处方数据格式参考 |
| WS/T 469 | 老年人健康管理技术规范 | 慢病管理 | 运动数据用于健康管理需符合 |
| WS/T 484 | 健康管理师国家职业标准 | 健康管理服务 | 健康管理师使用健身数据的合规要求 |
| WS/T 529 | 远程医疗信息系统基本功能规范 | 远程健康 | 运动数据远程传输接口规范 |

**关键说明**: 目前**没有专门针对健身器械数据**的GB/T或WS/T标准——这是市场空白，也是机会。HealthHub如果能推动建立"健身设备运动数据标准"，将建立行业话语权。

## 二、HL7 FHIR：国际健康数据互操作标准

### 2.1 什么是HL7 FHIR

HL7 FHIR (Fast Healthcare Interoperability Resources) 是国际医疗数据互操作的事实标准，由美国HL7组织维护。

- **全球采用率**: 2025年全球超过95%的主要医疗系统（包括美国、中国、欧盟）已采用或计划采用FHIR
- **中国进展**: 国家卫健委2022年发布《关于推进医疗健康信息互通共享的实施意见》，明确推动FHIR在中国的落地
- **Apple Health**: 已支持FHIR格式导出，苹果手表数据可直接导入医院系统

### 2.2 FHIR资源模型与健身数据映射

FHIR将健康数据抽象为"资源"(Resources)。以下是健身器械数据的FHIR映射：

| 健身数据类型 | FHIR资源 | 说明 |
|------------|----------|------|
| 运动记录（时间、时长、动作） | `Exercise` 或 `ActivityDefinition` | 记录单次运动 |
| 运动处方/计划 | `CarePlan` | 结构化运动处方 |
| 身体指标（心率、步数、卡路里） | `Observation` | 生命体征/测量数据 |
| 用户档案 | `Patient` | 用户基本信息 |
| 设备信息 | `Device` | 健身器械设备注册 |
| 运动评估报告 | `DiagnosticReport` | 训练质量评估输出 |

**对厂家的价值说明**:
> "如果您的设备数据采用FHIR格式，未来可以无缝对接到医院HIS系统、体检中心、健康管理平台。就像USB接口一样——符合标准的数据，走到哪都能用。"

### 2.3 FHIR在中国的落地情况

- **国家医疗健康大数据中心**: 已在福建、山东等省试点FHIR数据互通
- **医院互联互通评测**: 2023年起，三级医院互联互通评测（4甲以上）增加FHIR评测要求
- **中国FHIR推进委员会**: 2024年成立，推动FHIR本地化实施

## 三、HealthHub数据标准的现在与未来路线

### 3.1 当前数据层架构

HealthHub设备端SDK采集的数据格式设计原则：

```
IMU原始数据 → 设备端SDK处理 → 标准JSON封装
                                  ↓
                         字段映射至:
                         - 本地FHIR Observation模型
                         - 兼容WS/T元数据规范
                         - 支持GB/T 35273隐私合规
```

### 3.2 互联互通三层策略

| 层级 | 标准 | 实现方式 | 时间线 |
|------|------|---------|--------|
| 设备层 | BLE 5.0 + 自有协议 | nRF52840模组，TLS加密传输 | 已实现 |
| 平台层 | HL7 FHIR R4 | 数据存储按FHIR资源模型 | 2026 Q3 |
| 互通层 | WS/T + 医院HIS | API对接医院健康档案平台 | 2026 Q4-2027 |

### 3.3 为什么要提前对齐FHIR

**近期价值**:
- 体检机构（美年、爱康）正在建立健康数据互通标准，接入其体系需要FHIR格式
- 保险公司（平安、泰康）的"运动健康险"产品要求设备数据格式标准化
- 出口欧美时，FHIR格式数据更易被对方系统接受（GDPR合规）

**战略价值**:
- 成为"健身数据标准"的制定者，而非跟随者
- 宁津县400+厂家的数据聚合 + FHIR标准 = 行业数据交换协议的话语权

## 四、BD话术库

### 场景1: 厂家问"你的数据格式是什么？能不能换成别的？"

> "我们按HL7 FHIR国际标准和国内WS/T卫生行业标准设计数据层。数据格式是开放的——您随时可以导出原始数据，用其他平台分析。跟我们合作不是锁死，是合规接入整个健康数据生态。"

### 场景2: 厂家担心"数据主权"

> "GB/T 35273《个人信息安全规范》和PIPL都要求数据主权清晰。您的设备数据归属您，我们签合同白纸黑字保证不爬取、不碰数据、不用于其他客户。数据合规这块，我们比您更怕出事。"

### 场景3: 医院/健康管理机构问"能不能对接我们的系统"

> "HL7 FHIR是全球医疗数据互通标准，美国、中国三甲医院都在用。我们的平台支持FHIR格式导出，您医院的HIS系统只要也支持FHIR，就能直接对接。这是行业大趋势，我们提前做好了准备。"

### 场景4: 出口场景（欧洲）

> "GDPR要求数据处理有合法依据。我们建立了合规的数据采集框架，数据来源透明、处理有据、可导出删除——完全符合GDPR数据可携带权(Article 20)。这对欧洲买家是加分项。"
