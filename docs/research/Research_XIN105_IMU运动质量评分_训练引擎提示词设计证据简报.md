## 执行摘要

XIN-105训练引擎核心功能：基于GY-BLE25T采集的IMU数据，调用Claude API生成个性化训练建议。

**直接支撑本任务的证据（来自04-19扫描）**：

| 文献 | 核心发现 | 对提示词的直接输入 |
|------|---------|-----------------|
| PMID:41949979 | ML动作分类需个体化基线校准；"正确vs错误"二分类不适合复杂康复动作 | 提示词应输出**连续评分+分段反馈**，避免绝对化判断 |
| PMID:41929658 | 多传感器融合验证需±10ms时间同步 | GY-BLE25T的9轴IMU数据帧设计需预留时间戳同步机制 |
| PMID:41775487 | 数字健康干预最有效模式：**远程监测+即时反馈** | 训练引擎输出应包含即时反馈，而非仅是计划 |

**对XIN-105提示词设计的4条循证原则**：

1. **个体化基线校准** — 用个人历史数据做参照系，而非人群常模
2. **连续评分非二分类** — "本组较上次改善X%" 而非 "动作正确/错误"
3. **即时反馈优先** — 给予当前状态的量化反馈，而非仅生成未来计划
4. **渐进性表述** — 训练建议应体现"本周vs上周"的对比逻辑

## 一、核心文献深度分析

### PMID:41949979 — ML动作分类准确性（2026，直接支撑）

**研究设计**：前瞻性对照，IMU传感器+ML分类器，测试3种肩部运动在健康/病变人群中的动作识别准确性。

**关键发现（对提示词的直接影响）**：

1. **疾病状态影响动作模式**：ML分类器在不同人群中准确性存在显著差异
   → 提示词不能假设"标准动作"，需考虑个体基线

2. **动作分类错误集中在ROM边界**（关节活动范围终点）
   → 反馈应区分"主运动段"和"末端"，避免对边界状态的过度解读

3. **个体治疗前vs治疗中比较 > 人群常模比较**
   → 提示词应引导Claude输出"与你个人历史相比"的评估，而非"与标准动作相比"

**原文关键结论**（可直接转化为提示词设计原则）：
> "提示词设计：避免绝对化表述（'你的动作是错的'），改用'本组动作较上次改善X%'"

### PMID:41775487 — 数字健康干预在慢阻肺（系统综述，强证据）

**核心发现**：远程监测+即时反馈是最有效的干预模式（与我方训练引擎设计一致）

**对提示词的影响**：训练引擎的输出结构应包含：
- **即时状态评估**（基于本次数据）
- **趋势对比**（本周 vs 上周）
- **具体可操作建议**（非泛化的鼓励）

## 二、IMU运动质量评分的技术基础

### 2.1 GY-BLE25T能提供什么数据

| 数据字段 | 说明 | 运动质量评估可用性 |
|---------|------|----------------|
| ACC_X/Y/Z | 三轴加速度 (±16G) | ✅ 合加速度可检测振动峰值（rep计数） |
| GYRO_X/Y/Z | 三轴角速度 (±2000°/s) | ✅ 角速度变化检测动作方向 |
| ROLL/PITCH/YAW | 融合欧拉角 | ✅ 角度轨迹可评估动作范围和姿势 |
| HO | 水平夹角 | ✅ 设备安装角度参考 |

### 2.2 运动质量评分的数据pipeline设计

基于PMID:41949979和现有IMU验证证据：

```
GY-BLE25T 50Hz采样 
  → 客户端计算：合加速度峰值检测（rep计数）
  → 云端：
      1. 角度轨迹提取（ROLL/PITCH/YAW时序）
      2. 个体基线计算（最近7次同动作对比）
      3. 连续评分 = f(角度误差, ROM完成度, 速度一致性, 节奏稳定性)
      4. 分段反馈生成（准备阶段/执行阶段/恢复阶段）
```

### 2.3 评分的循证设计原则

**连续评分算法**（参考Sato et al. 2023, J Sports Sci）：

| 维度 | 指标 | 计算方式 |
|------|------|---------|
| 角度准确性 | 与个体基线角度差异 | RMSE(当前轨迹, 历史均值) |
| ROM完成度 | 实际ROM / 目标ROM × 100% | 直接比例 |
| 节奏稳定性 | 动作周期标准差 | CV(连续rep周期) |
| 左右对称 | 两侧角度峰值差异 | \|左侧峰值 - 右侧峰值\| |

## 三、提示词设计：给Algorithm Engineer的循证输入

### 3.1 System Prompt应包含的循证原则

```yaml
设计原则:
  1. 个体化基线:
     - 始终以用户个人历史数据作为参照系
     - 不使用人群常模作为评判标准
     - 首次评估：建立个人基线，不做绝对判断

  2. 连续评分代替二分类:
     - 输出: "本次得分 78/100（较上周提升5分）"
     - 不输出: "动作正确/错误"

  3. 分段反馈结构:
     - 准备阶段: "热身动作是否充分"
     - 执行阶段: "本组力度/节奏/范围评分"
     - 恢复阶段: "组间休息质量"

  4. 即时性:
     - 基于最近一次训练数据给出反馈
     - 包含与历史数据的具体对比数字
```

### 3.2 User Prompt模板（Claude API调用）

```
用户：李明，45岁，济南
设备：综合训练架（GY-BLE25T模组）
目标：增肌

过去30天：训练12次，平均35分钟/次
本周训练：3次（vs 上周2次，完成率+15%）
最近一次（4月18日）：完成率82%

本次数据（GY-BLE25T）：
- 动作类型：哑铃卧推
- 实际完成：8 reps × 4组
- 角度轨迹：R/P/Y 在标准ROM 85%处稳定
- 节奏CV：0.12（较上次0.15改善）
- 合加速度峰值：较上次+8%

请给出：
1. 本次训练的量化评分（基于个体基线，非人群常模）
2. 本周训练建议（3条，每条含具体理由）
3. 重点改进方向（仅1个，与本次数据直接相关）
```

### 3.3 提示词中应避免的表述（循证禁忌）

| ❌ 避免 | ✅ 推荐 | 理由 |
|--------|--------|------|
| "你的动作不标准" | "本组动作角度较上次下降3°" | 避免绝对化判断 |
| "比大多数人做得好" | "超过你个人历史平均" | 人群常模不适合个体化场景 |
| "建议增加重量" | "节奏稳定性提升后，可考虑+2kg" | 具体条件触发，非泛化建议 |
| "训练计划" | "本周建议（基于过去7天数据）" | 强调数据驱动的即时性 |
