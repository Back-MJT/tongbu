- 阻力调节：电磁控阻（非传统摩擦）

- 定价：¥3,999（旗舰款）
- 智能化水平：内容-硬件联动初级，闭环反馈依赖内容团队预设参数

**Keep Cardio Flex（2025年）**
- 新功能：AI动作识别摄像头（手机App端）
- 技术路线：纯视觉方案（非可穿戴/IMU）
- 限制：仅支持手机拍摄场景，无法嵌入健身器械

**Keep Watch 智能手表（2024年迭代款）**
- 心率监测：升级为8通道光学心率（vs 4通道）
- 定位：运动专项手表（非医疗）
- BLE开放：部分数据接口对Keep App开放，不对第三方硬件开放

**Keep Business 企业版（持续迭代）**
- 新增：团队竞赛功能（数据驱动团队PK）
- 数据看板：基础运动数据聚合，无深层健康干预
- API状态：不对外开放，运动数据仅限Keep生态内使用

**技术评估**:
| 技术维度 | Keep现状 | 对昕动影响 |
|---------|---------|-----------|
| 设备接入 | 自营封闭，仅Keep硬件 | 合作机会（Keep内容×昕动硬件） |
| 干预引擎 | 弱，仅基础课程推荐 | 差异化空间大 |
| B2B能力 | 一般，企业功能有限 | 昕动专注B2B设备数据，更有深度 |
| BLE开放性 | 封闭生态，无第三方接入 | 昕动中立平台优势明显 |
| IMU/传感器 | 依赖手机视觉，无健身器械IMU | 昕动在健身器械IMU场景有先发优势 |

**BD话术**: "Keep的闭环在内容层，昕动的闭环在设备层——我们接的是健身器械的'出厂级'数据，不是手机App的运动记录。"

## 二、微信小程序BLE API能力报告（v2.0，基于XIN-96验证）

### 2.1 BLE API完整能力清单

微信小程序BLE API（wx.bluetoothBLE）是微信小程序连接物理设备的**唯一官方通道**：

#### 核心API能力

| API | 功能 | 对GY-BLE25T的适用性 |
|-----|------|-------------------|
| `wx.openBluetoothAdapter()` | 初始化蓝牙适配器 | ✅ 必需 |
| `wx.closeBluetoothAdapter()` | 关闭蓝牙适配器 | ✅ 善后处理 |
| `wx.getBluetoothAdapterState()` | 获取适配器状态 | ✅ 状态检查 |
| `wx.startBluetoothDevicesDiscovery()` | 开始扫描周围设备 | ✅ 扫描IMU |
| `wx.stopBluetoothDevicesDiscovery()` | 停止扫描 | ✅ 节省电量 |
| `wx.getBluetoothDevices()` | 获取已发现设备列表 | ✅ 缓存设备 |
| `wx.createBLEConnection()` | 连接BLE设备 | ✅ 连接IMU |
| `wx.closeBLEConnection()` | 断开连接 | ✅ 善后处理 |
| `wx.getBLEDeviceServices()` | 获取设备服务列表 | ✅ 服务发现 |
| `wx.getBLEDeviceCharacteristics()` | 获取服务特征值 | ✅ 特征发现 |
| `wx.readBLECharacteristicValue()` | 读取特征值 | ❌ IMU不用 |
| `wx.writeBLECharacteristicValue()` | 写入特征值 | ✅ 配置IMU参数 |
| `wx.notifyBLECharacteristicValueChange()` | 启用特征值通知 | ✅ 接收IMU数据 |
| `wx.onBLECharacteristicValueChange()` | 监听特征值变化 | ✅ **核心数据接收** |
| `wx.onBluetoothDeviceFound()` | 监听新设备发现 | ✅ 实时发现 |

#### 关键限制

| 限制项 | 说明 | 对IMU场景影响 |
|--------|------|-------------|
| **iOS限用CoreBluetooth** | iOS必须走Apple私有API，需企业账号 | ⚠️ **重大风险**：iOS无法使用微信小程序连接BLE设备 |
| **Android碎片化** | 不同厂商蓝牙栈实现差异大 | 中（需多机型测试）|
| **单次连接设备数** | 建议不超过3个 | ✅ 够用（1个IMU） |
| **MTU大小** | 默认23字节，有效载荷20字节 | ⚠️ 需处理22字节分帧 |
| **后台运行限制** | 息屏后可能被系统中断 | 中（健身场景用户一般不锁屏）|
| **扫描功耗** | 扫描时电流约10-15mA | ✅ 健身器械有供电 |

#### 微信基础库版本要求

| 功能 | 最低基础库版本 |
|------|-------------|
| 核心BLE API | 1.1.0+（已覆盖99%用户）|
| `wx.setBLEMTU()` | 2.20.1+（解决22字节分帧问题）|
| 获取设备信号强度(RSSI) | 2.7.0+ |

---

### 2.2 iOS问题详细说明（最重要风险）

**问题**: 微信小程序在iOS上使用BLE时，Apple要求必须使用CoreBluetooth私有API，而微信小程序的BLE API在iOS上**实际上不工作**。

**验证状态**: 微信官方文档在"常见问题"中已注明iOS BLE兼容性差异，但未明确说明完全不可用。

**实测参考**:
- 微信运动手环（如小米手环）: iOS可用 → 说明**某些设备通过了苹果MFi认证**绕过了限制
- 第三方BLE设备: 大多数在iOS上失败

**解决方案（3条路径）**:

| 方案 | 描述 | 适用场景 | 实施难度 |
|------|------|---------|--------|
| **方案A: iOS使用微信硬件准入** | 通过微信硬件平台（微信硬件公众号）获取iOS BLE白名单 | 已获得微信认证的硬件 | 高（需微信合作）|
| **方案B: iOS降级到BLE广播模式** | iOS微信小程序接收BLE广播数据（Beacon模式） | 仅接收IMU数据，不写入 | 中（功能受限）|
| **方案C: iOS用户提供账号密码登录** | iOS用户通过手机蓝牙直连器械，器械再通过WiFi上传数据 | 牺牲部分实时性 | 低（需后台架构配合）|

**推荐**: MVP阶段，**Android为主**，iOS通过方案C降级支持（不要求BLE直连，数据通过器械内置WiFi上云）

### 2.3 GY-BLE25T × 微信小程序兼容性矩阵

基于XIN-96验证报告（2026-04-14）:

| 能力 | 微信API | GY-BLE25T | 兼容性 | 备注 |
|------|---------|----------|--------|------|
| 设备扫描 | `startBluetoothDevicesDiscovery` | 广播`gy_ble25t1` | ✅ | 精确匹配设备名 |
| 建立连接 | `createBLEConnection` | 标准BLE | ✅ | Android正常 |
| 服务发现 | `getBLEDeviceServices` | Service FFE0 | ✅ | UUID标准格式 |
| 特征发现 | `getBLEDeviceCharacteristics` | FFE1~FFE4 | ✅ | 全部4个特征 |
| 数据通知 | `notifyBLECharacteristicValueChange` | FFE4 Notify | ✅ | 订阅IMU数据 |
| 数据接收 | `onBLECharacteristicValueChange` | 22字节ArrayBuffer | ✅ | 需int16解析 |
| 参数写入 | `writeBLECharacteristicValue` | FFE1/FFE3 Write | ✅ | 配置更新速率/量程 |
| MTU协商 | `setBLEMTU` (2.20.1+) | 22字节<MTU | ⚠️ | 微信≥2.20.1才支持 |

**结论**: 技术上**完全可行**，iOS是主要风险点（与微信BLE无关，是Apple限制）。

### 2.4 BLE开发最佳实践（技术团队参考）

#### 连接流程状态机

```
IDLE → OPENING → OPENED → CONNECTING → CONNECTED → READY → DISCONNECTED
         ↓          ↓         ↓           ↓          ↓
      ERROR(1)  ERROR(2)   ERROR(3)     ERROR(4)    ERROR(5)
```

#### 错误处理策略

```javascript
// 错误码处理
const BLE_ERRORS = {
  10000: '未初始化蓝牙适配器 → 调用 openBluetoothAdapter()',
  10001: '蓝牙适配器不可用 → 检查手机蓝牙开关',
  10002: '设备服务不存在 → 确认GY-BLE25T固件版本',
  10003: '连接失败 → 重新扫描并连接',
  10004: '特征值不存在 → 检查服务UUID和特征UUID',
  10005: '连接已断开 → 重新建立连接',
  10006: '设备不存在 → 设备已离开范围',
}
```

#### 电量优化建议

1. **按需扫描**: 用户主动点击"连接"时才扫描，不做持续后台扫描
2. **连接后停止扫描**: `wx.stopBluetoothDevicesDiscovery()` 在连接成功后立即调用
3. **息屏保活**: Android可保持连接，健身期间一般不锁屏
4. **断开时机**: 运动结束页面离开时主动断开 (`wx.closeBLEConnection()`)

## 三、健身器械数字化案例集（国内外参考）

### 3.1 国际案例

#### Technogym（意大利，健身器械顶级品牌）

- 产品：跑步机、划船机等全线器械 + Mywellness云平台
- 数字化功能：每台器械内置传感器，记录训练数据
- 数据能力：用户扫描器械二维码，数据同步至App
- 干预引擎：无（纯数据记录）
- 商业模式：硬件销售 + SaaS订阅
- 与昕动关系：潜在合作或学习对象；他们的"设备数据+云"架构与昕动四层产品类似，但无AI干预层

#### Life Fitness（美国，商用健身器械龙头）

- 产品：商用跑步机、椭圆机等（Hammer Strength品牌）
- 数字化：Discover HD Console（触控屏，带App集成）
- 技术路线：开放API给第三方（器材+软件生态）
- 与昕动关系：国内宁津县代工厂的目标标杆；BD话术可引用"连Life Fitness都在做数字化"

#### Peloton（美国，联网健身典型）

- 商业模式：硬件（跑步机/单车¥2-4万）+ 会员订阅（¥400/月）
- 技术亮点：座椅/踏板内置传感器，实时捕捉骑行功率、踏频、姿势
- 干预引擎：教练实时口令调整（真人+AI教练），内容-硬件联动
- 与昕动关系：内容-硬件联动思路一致，但Peloton是封闭生态；昕动可以"无 Peloton 的硬件，接入更丰富的内容"

### 3.2 国内案例

#### 英派斯（国内商用健身器械上市公司）

- 产品：商用跑步机、力量器械，出口为主
- 数字化：早期探索（无成熟方案）
- 与昕动关系：潜在P1客户

- 模式：麦瑞克跑步机接入华为鸿蒙，用户用华为健康App查看数据
- 意义：国内传统健身器械首次规模化接入手机生态
- 对昕动启发：华为能做到的是"数据展示"，昕动要做到的是"数据→处方→行为改变"

### 3.3 数字化成熟度评估框架

用于评估潜在B2B客户（厂家）的数字化现状：

| 维度 | L1: 无数字化 | L2: 基础记录 | L3: 数据联网 | L4: 智能分析 | L5: AI干预 |
|------|------------|------------|------------|------------|-----------|
| 数据采集 | 无 | 人工记录 | 传感器自动 | 多传感器融合 | 传感器+AI |
| 数据传输 | 无 | 本地存储 | WiFi/BLE上云 | 实时上云 | 实时+低延迟 |
| 数据应用 | 无 | 历史查看 | 基础统计 | 健康评分 | 个性化处方 |
| 代表品牌 | 传统小厂 | 舒华App | 麦瑞克×华为 | 暂无 | **昕动（目标）** |

**BD话术**: "宁津县大多数健身器械厂家目前处于L1-L2阶段——数字化是蓝海。昕动帮你从L1直接跳到L4，不需要自己建团队。"
