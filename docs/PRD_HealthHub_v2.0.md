## 0. 执行摘要

HealthHub v2.0 是适配 Board v1.1 战略转向的完整产品需求文档，定义健身器械行业数字神经系统的四层产品体系：**IMU+BLE硬件模组(第零层) → 设备数字化SDK(第一层) → AI训练引擎(第二层) → 数据价值(第三层)**。

**目标市场**: 山东省德州市宁津县 — 400+家健身器械厂商
**核心差异**: 第零层IMU+BLE模组让传统抗阻器械产生数字信号，填补行业空白
**用户端**: 微信小程序为唯一C端触点
**AI引擎**: Claude API，用户阶段识别 + 个性化训练方案生成
**变现路径**: 厂家年费(¥8,000-20,000/年) + 用户订阅(¥29-49/月) + 数据服务

## 2. 第零层：IMU+BLE硬件模组层（核心差异化）

### 2.1 战略定位

传统抗阻器械是宁津县出货量最大的品类（占整体60%+），却全部是哑终端。IMU+BLE模组是让这批器械产生数字信号的唯一方式，也是本平台对比纯软件竞争对手最难复制的技术壁垒。

**为什么是这里**：纯软件SDK方案无法覆盖无数字接口的传统抗阻器械；IMU+BLE模组贴装框架3分钟完成安装，不改变生产工序。

### 2.2 硬件规格

| 组件 | 选型 | 理由 |
|------|------|------|
| IMU模组 | **GY-BLE25T**（国产9轴一体模组） | 6轴加速度+陀螺仪 + 融合算法输出欧拉角 + BLE 5.0，自带固件无需自研嵌入式代码 |
| BLE | 内置于GY-BLE25T | BLE 5.0 GATT连接，FFE0 Service / FFE4 Notify，微信小程序直接兼容 |
| 供电方案 | 框架取电（优先）/ CR2032纽扣电池（备选，寿命12-18个月） | 零换电维护成本 |
| 防护等级 | IP54 | 健身房环境足够，成本低于IP67 |
| BOM成本 | 待确认（模组已采购，成本低于nRF52840+外挂IMU方案） | 厂家可接受范围，换取AI智能卖点 |
| 安装方式 | 热缩螺丝固定至框架指定位置 | 工人操作3分钟，不改变生产工序 |

> **⚠️ 硬件变更说明 (2026-04-15, XIN-111)**: 原方案为Nordic nRF52840 + ST LSM6DSO（需自研Zephyr固件），现已替换为国产GY-BLE25T成品模组（XIN-96已验证）。GY-BLE25T自带传感器融合固件，通过BLE GATT直接输出处理后的欧拉角/加速度/陀螺仪数据（22字节/帧），大幅降低开发复杂度。详见 `docs/GY-BLE25T_IMU模块验证报告.md`。

### 2.3 可采集数据

| 数据类型 | 采集方式 | 精度 | 用途 |
|----------|----------|------|------|
| 使用检测（有人/无人） | 加速度阈值触发 | 99%+ | 厂家后台：设备实时使用率 |
| 组数计数（rep counting） | 振动峰值检测+30秒静默分组 | <5%误差（标准动作） | 用户训练记录：今日X组 |
| 使用时长 | 有效振动时间累计 | 精确 | 训练量统计，AI引擎输入 |
| 大概重量等级 | 振动强度代理估算（轻/中/重） | 低精度，仅供参考 | 训练强度趋势分析 |
| 器械空闲时段 | 无振动时间统计 | 精确 | 健身房运营：高峰低谷分析 |

### 2.4 组数计数算法（固件级）

```
采样频率：50Hz（省电，足够捕捉抗阻动作）
合加速度计算：√(ax² + ay² + az²)

if (合加速度 > 阈值1.5g) {
  if (距上次rep > 500ms) {  // 抖动过滤
    rep_count++;
    last_rep_time = now;
  }
}

// 组间判断：超过30秒无rep → 本组结束，记录到云端
if (now - last_rep_time > 30s) {
  upload_set(rep_count, duration, intensity);
  rep_count = 0;
}
```

### 2.5 数据上传方案（双模式）

| 模式 | 适用场景 | 技术方案 | 阶段 |
|------|----------|----------|------|
| 用户手机直连 | 家用/小型健身房 | 微信小程序通过BLE接收模组数据，训练结束后上传云端 | Phase 1 — 优先 |
| 固定WiFi网关 | 商用健身房 | 树莓派Zero(¥80)自动收集所有模组数据，不依赖用户手机 | Phase 2 — 有商用客户后 |

### 2.6 OEM生产线集成流程

**厂家最关心：要改多少生产流程？答案是：几乎不改。**

- 原有流程：焊接 → 组装 → 调试 → 出厂
- 新增步骤：组装完成后，工人用热缩螺丝固定模组到框架指定位置（2分钟），扫描模组二维码绑定设备ID（1分钟）
- 不需要额外设备，不需要改流水线，不需要专项培训，工人当天可上手

## 3. 第一层：设备数字化层（Device SDK）

### 3.1 功能范围

- **设备注册与绑定**：每台设备生成唯一数字身份（Device ID），出厂或首次联网时完成注册
- **数据采集**：运动时长、使用次数、运动强度（通过IMU传感器）、完成率
- **厂家管理后台**：Web端Dashboard，展示名下设备在线状态、今日/本周使用数据、活跃用户趋势
- **固件OTA通道**：为后续功能更新保留远程升级能力

### 3.2 技术路线

- **目标设备**：优先对接已有中控屏的Android系统设备（占现有设备20-30%）
- **接入方式**：Android SDK嵌入，与厂家固件团队深度合作
- **数据传输**：MQTT协议上云，阿里云IoT Platform托管
- **状态**：IMU+Android App+Web后台完整链路Demo已验证

### 3.3 厂家管理后台页面设计

**路径**: 现有 b2b-frontend 改版，数据源从健康机构切换为健身器械厂家

| 页面 | 核心内容 |
|------|----------|
| 设备列表 | 所有设备在线状态（绿/黄/红）、型号、安装日期 |
| 今日概况 | 今日总训练人次、使用设备数、活跃时段热力图 |
| 使用趋势 | 近7天/30天使用次数趋势折线图 |
| 用户阶段分布 | 接入用户中：初学/成长/停滞/进阶 各多少人 |
| 流失预警 | 连续7天未训练的用户列表（设备维度） |
| 固件管理 | OTA升级状态、版本列表 |

### 3.4 设备绑定流程

```
用户：打开微信小程序 → 扫描设备二维码 → BLE连接配对 → 绑定成功
数据：设备 → BLE → 手机小程序 → MQTT → 云端
```

## 4. 第二层：AI训练引擎层（Personalization Engine）

### 4.1 设计原则

引擎不替代厂家现有App，而是以模块形式嵌入，输出个性化内容。所有AI能力通过调用Claude API实现，无需自研模型。

### 4.2 用户阶段识别

| 用户阶段 | 判断依据 | 引擎行为 |
|----------|----------|----------|
| 初学期 (0-4周) | 使用频率<3次/周，完成率<60% | 降低难度建议，强化正向反馈，成就解锁激励 |
| 成长期 (1-3个月) | 使用稳定，完成率>70% | 渐进式负荷增加，引入多样化训练 |
| 停滞期 | 连续7天未使用，或强度无增长 | 流失预警推送，调整计划难度 |
| 进阶期 (3个月+) | 高频使用，主动提高强度 | 周期化训练方案，引入竞技目标 |

### 4.3 Claude API调用规范

- temperature设为0，保证同样输入输出一致
- 所有调用经统一callClaude()函数路由，记录完整日志（prompt内容、返回内容、token消耗、延迟、成本）
- 引入缓存层：同一用户数据未变化时直接返回缓存结果，减少API调用成本
- System Prompt单独版本管理，与业务代码解耦，支持A/B测试

### 4.4 个性化建议输入模板

```
用户：{姓名}，{年龄}岁，{性别}，{城市}
设备：{器械类型}，型号：{型号}
目标：{运动目标}
过去30天：训练{N}次，平均{M}分钟/次
上周训练：{X}次（vs 上上周{Y}次）
最近一次：{日期}，完成率{Z}%
当前阶段：{阶段判断}

请用中文给出本周训练建议（3条，每条含具体理由，语气像私教朋友，不要套话）。
```

## 6. OEM生产线集成操作手册框架

### 6.1 手册目的

指导宁津县健身器械厂家在现有生产线上集成IMU+BLE模组，实现设备数字化。目标：**不改变生产工序，3分钟完成单台模组安装**。

### 6.2 集成前准备

| 准备项 | 说明 | 负责人 |
|--------|------|--------|
| 模组到货检验 | 检查模组外观、电池固件版本 | 采购/QC |
| 设备型号匹配 | 确认模组与目标器械型号匹配 | 技术对接人 |
| 二维码印刷 | 设备专属二维码（由HealthHub平台生成） | HealthHub提供 |
| 员工培训 | 15分钟操作培训，观看安装视频 | HealthHub提供材料 |

### 6.3 标准安装流程（3分钟）

**第一步：模组固定（2分钟）**
1. 找到器械框架指定安装位置（见各型号安装示意图）
2. 用热缩螺丝将模组固定在框架上
3. 确保模组外壳缺口对准框架凹槽

**第二步：设备绑定（1分钟）**
1. 用扫码枪扫描模组外壳上的二维码
2. 在生产厂家系统输入设备型号和序列号
3. 系统返回绑定成功确认

**注意**：无需焊接，无需打孔，不影响原有装配流程。

### 6.4 安装检查清单

- [ ] 模组固定螺丝拧紧，无松动
- [ ] 模组外壳与框架紧密接触（利于框架取电）
- [ ] 二维码扫描绑定成功
- [ ] 模组LED指示灯亮起（绿色=正常）

### 6.5 常见问题

| 问题 | 原因 | 解决方案 |
|------|------|----------|
| LED不亮 | 电池未激活或电量耗尽 | 静置2分钟激活，或更换CR2032电池 |
| 扫码失败 | 二维码损坏或印刷模糊 | 手动在系统输入序列号 |
| 数据未上报 | 手机蓝牙未开启 | 引导用户开启蓝牙 |

### 6.6 质量控制节点

- 产线终端抽检比例：每50台抽检1台
- 抽检项目：模组固定牢固度、二维码可扫描性、信号测试
- 不合格处理：返工重新安装，整批复检

## 8. 技术约束

### 8.1 技术栈

| 层级 | 技术选型 |
|------|----------|
| 硬件模组 | GY-BLE25T（国产9轴IMU+BLE 5.0一体模组，自带固件，无需自研嵌入式代码） |
| 设备端SDK | Android (Kotlin/Java) + IMU数据采集库 |
| 数据传输 | MQTT over TLS → 阿里云IoT Platform |
| 后端服务 | Node.js / Python (FastAPI) |
| 数据库 | PostgreSQL（结构化数据）+ TimescaleDB（时序数据） |
| 缓存层 | Redis（Claude API响应缓存） |
| AI引擎 | Claude API (claude-sonnet-4-20250514, temperature=0) |
| 用户端 | 微信小程序（Taro框架） |
| 厂家后台 | React + Ant Design Pro（现有前端改版） |
| 云基础设施 | 阿里云（国内合规） |

### 8.2 PIPL合规要点（从第一天执行）

- 用户注册时强制展示隐私协议，明确说明采集数据类型、用途（含聚合分析）
- 用户ID与真实身份信息分离存储，聚合分析只使用匿名化数据
- 厂家合作协议明确：原始数据归平台所有（匿名聚合后），厂家有权查看自身设备数据
- 数据保留策略：原始数据保留3年，超期自动归档压缩
