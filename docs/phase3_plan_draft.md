## 2. Phase 3 战略重心

> **从 B2B 到 B2B2C：通过机构网络触达终端消费者，建立用户规模和收入。同步启动 2G（政府/医保）场景探索。**

### 核心假设
- B2B 机构客户 ≥3 家，完成技术对接
- 干预引擎在运动/睡眠场景有真实用户数据验证
- 种子资金支撑 12 个月运营

## 3. 工作分解

### 3.1 C端产品 MVP

**目标**: 触达终端消费者，建立用户规模和留存数据

#### 3.1.1 C端 APP / 小程序

| 功能 | 优先级 | 描述 |
|------|--------|------|
| 个人健康看板 | P0 | 设备数据可视化、运动/睡眠趋势 |
| 每日干预任务 | P0 | 推送个性化运动处方、提醒 |
| 进度追踪 | P0 | 干预依从性可视化、成就感系统 |
| 健康师咨询入口 | P1 | 接入 B2B 健康师工作台 |
| 社交分享 | P2 | 邀请有礼、朋友圈分享 |
| 健康资讯 | P2 | 基于用户画像的个性化内容推荐 |

**MVP 边界**: 仅支持微信小程序；仅支持运动/睡眠 2 个干预场景；不包含支付

#### 3.1.2 用户增长体系

| 功能 | 优先级 | 描述 |
|------|--------|------|
| 邀请码系统 | P1 | 机构分配邀请码，C 端注册时关联 |
| 家庭账号 | P2 | 一人账号可关联家庭成员 |
| 成就徽章 | P2 | 干预坚持天数徽章 |

---

### 3.2 规模化架构

**目标**: 支持万级设备同时在线，支撑 B2B2C 增长

| 功能 | 优先级 | 描述 |
|------|--------|------|
| 高并发数据管道优化 | P0 | 万级设备同时在线，延迟 <500ms |
| 多租户架构全面完善 | P0 | 租户配额管理、计费层 |
| 边缘计算节点部署方案 | P1 | 区域化部署，降低延迟 |
| 数据分片与水平扩展 | P0 | PostgreSQL 分片 + 时序 DB 扩展 |
| SLA 保障与灾备机制 | P1 | 99.5% 可用性目标 |

---

### 3.3 开放平台

**目标**: 聚合行业需求，建立生态壁垒

| 功能 | 优先级 | 描述 |
|------|--------|------|
| 第三方设备接入 API | P1 | SDK + 开发者文档，支持更多设备 |
| 第三方服务市场 | P2 | 营养咨询、运动指导等增值服务接入 |
| 数据服务 | P2 | 脱敏数据服务（研究机构合作） |

---

### 3.4 合规与认证

**目标**: 满足医疗健康数据合规要求，为 B2B2C 扫清监管障碍

| 事项 | 优先级 | 描述 |
|------|--------|------|
| 个人信息保护法合规 | P0 | 数据采集知情同意、隐私政策、账号注销权 |
| 医疗器械相关资质评估 | P1 | 评估是否需要二类医疗器械证 |
| 信息安全三级等保 | P1 | 数据库加密、访问审计 |
| 算法伦理审查机制 | P2 | 干预推荐透明化、用户可拒绝 |

---

### 3.5 标准制定

**目标**: 掌握行业标准话语权，建立竞争壁垒

| 事项 | 优先级 | 描述 |
|------|--------|------|
| 健康装备数据接入规范白皮书 | P1 | 发布统一设备接入规范 |
| 参与行业/团体标准起草 | P2 | 争取成为标准制定参与者 |
| 开放设备适配层 SDK | P2 | 降低第三方设备接入成本 |

## 6. 关键风险

| 风险 | 影响 | 应对 |
|------|------|------|
| C端获客成本高 | 高 | 依托 B2B 机构现有用户池，不单独投流 |
| 合规审批周期长 | 中 | 提前 6 个月启动资质申请 |
| C端留存率低 | 高 | 干预效果可见 + 健康师人工干预提升粘性 |
