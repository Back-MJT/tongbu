# B Line 小程序成熟版 V1 产品规范

## 1. 产品定位

**目标用户**: 健身场馆会员（主要是时间紧张的上班族，使用器械进行力量训练）
**核心价值**: 将传统健身器械智能化，实时展示训练完成次数、组数、时长，让用户无需低头看手机就能掌握训练进度。
**产品阶段**: V1 成熟化——蓝牙稳定、计数准确、UI专业、后台可用、A Line预留。

---

## 2. 目标场景

### 真实健身房使用场景
- 用户进入健身房 → 打开微信小程序 → 扫码器械二维码 → 连接传感器 → 开始训练
- 训练过程中抬头看小程序屏幕就能看到完成次数（不低头看手机）
- 训练结束 → 离开器械 → 传感器自动释放供他人使用
- 其他用户可随时扫码同一器械（如果上一用户已离开）

### 异常场景
- 用户A连接器械训练中，用户B扫码同一器械 → B看到"器械正被使用"
- 用户离开未主动结束训练（小程序崩溃/强退） → 5分钟内传感器自动释放
- 用户扫码后切换到另一台器械 → 旧传感器可靠断开，新连接不受影响

## 3. 核心指标定义

### 训练计数标准
| 指标 | 定义 | 采集方式 |
|------|------|----------|
| totalReps | 训练总次数 | 状态机完整 cycle 计数 |
| sets | 完成组数 | 25秒无动作自动切组 |
| duration | 总训练时长（秒） | 从首次有效计数到训练结束 |
| avgRepTime | 平均每次用时（秒） | duration / totalReps |
| sensorStatus | 传感器状态 | BLE连接状态 + IMU数据有效性 |

### 成熟版计数要求
- **防抖**: 相邻有效计数间隔 ≥ 600ms
- **防误**: 上下运动幅度差值 < 15° 不计入
- **防重**: 同组内相同相位不重复计数
- **半程误判**: 上下阈值分别为 20° 和 5°，防止半程被误计

## 5. 蓝牙连接规范

### 连接状态机
```
idle → scanning → connecting → connected → training → disconnecting → idle
```

### 可靠断开要求
- `disconnect()` 必须等待 `wx.closeBLEConnection()` 完成才能返回
- 页面 `onUnmounted` 时 disconnect 优先于 release 执行
- 切换器械时旧连接断开完成后才能开始新连接

### 防重复操作
- 扫码按钮点击后 2 秒内禁止再次点击（防重复扫码）
- 连接中状态禁止触发新的连接
- 断开连接过程中禁止新的连接请求

### 占用与心跳
- 占用后每 30 秒心跳一次
- 心跳失败立即显示重连提示
- 5 分钟无 IMU 数据自动断开（用户离开保护）

## 6. 异常状态规范

| 场景 | 前端表现 | 用户引导 |
|------|----------|----------|
| 蓝牙未开启 | 弹窗引导开启蓝牙 | "请开启蓝牙后重试" |
| 传感器无信号 | 实时页显示"信号丢失" | "请检查传感器电量" |
| 器械被占用 | 扫码后显示占用状态 | "器械正被使用，请稍后" |
| 网络失败 | Toast + 重试按钮 | "网络不佳，是否重试？" |
| 训练中断（大退） | 5分钟后自动释放 | 无需用户操作 |

## 8. 测试验收标准

### BLE P0 验收
1. A连接器械训练中，强退小程序 → B尝试连接同一器械 → 30秒内成功
2. A扫器械1 → 连接传感器 → A扫器械2二维码 → 传感器切换无延迟（< 2秒）
3. 用户离开页面 → 传感器立即释放（无IMU数据时5分钟自动断连）

### 计数 P0 验收
1. 标准深蹲测试：10次完整深蹲 → 显示 10 reps，误差 0
2. 快速连续动作：2秒内10次 → 实际计数 ≤ 3（防抖生效）
3. 半程误判：只蹲一半 → 不计入
4. 组间休息：25秒无动作 → 自动切组

### UX P1 验收
1. 实时计数页：完成次数大字显示（字号 ≥ 72px）
2. 训练结果页：跳转后 3 秒内展示数据
3. 进度页柱状图有数值标签

### 后台 P2 验收
1. 后台管理员3步内完成扫码绑定新器械
2. 器械状态变更在后台5秒内刷新
