## 2. IMU传感器选型对比

### 2.1 候选芯片对比

| 参数 | MPU6050 | BMI160 | ICM-42688 | LSM6DSO32 |
|------|---------|--------|-----------|-----------|
| **加速度计量程** | ±2/4/8/16g | ±2/4/8/16g | ±2/4/8/16/32g | ±2/4/8/16/32g |
| **陀螺仪量程** | ±250/500/1000/2000°/s | ±125/250/500/1000/2000°/s | ±16/32/64/128/256/512/1024/2048°/s | ±16/32/64/125/250/500/1000/2000°/s |
| **加速度噪声密度** | 400 μg/√Hz | 120 μg/√Hz | 23 μg/√Hz | 30 μg/√Hz |
| **陀螺仪噪声密度** | 0.005°/s/√Hz | 0.010°/s/√Hz | 0.0029°/s/√Hz | 0.003°/s/√Hz |
| **采样率** | ≤4kHz | ≤1.6kHz | ≤8kHz | ≤6.66kHz |
| **通信接口** | I2C, SPI | I2C, SPI | I2C, SPI | I2C, SPI |
| **BLE兼容模块** | GATT Custom | GATT Custom | GATT Custom | GATT Custom |
| **典型功耗** | 3.8mA (正常) | 3.0mA (正常) | 2.0mA (正常) | 2.5mA (正常) |
| **封装** | QFN-16 (3x3mm) | LGA-14 (2.5x3mm) | QFN-16 (3x3mm) | LGA-14 (2.5x3mm) |
| **价格 (批量)** | ¥3-5 | ¥8-12 | ¥15-20 | ¥6-10 |
| **成熟度** | 非常成熟 | 成熟 | 新一代 | 成熟 |

### 2.2 推荐方案

**首选: ICM-42688**

理由：
1. **最高精度**: 加速度噪声密度23 μg/√Hz，陀螺仪0.0029°/s/√Hz，能够检测细微运动变化
2. **宽量程**: 加速度最高±32g，陀螺仪最高±2048°/s，覆盖深蹲/硬拉/卧推等大冲击力场景
3. **高采样率**: 8kHz采样率，可捕捉快速动作细节
4. **低功耗**: 2mA典型功耗，延长电池寿命

**成本敏感场景备选: MPU6050**

理由：
- 价格极低（¥3-5/片）
- 性能足够基础力量训练识别（深蹲次数、组数）
- 社区支持丰富，驱动成熟

**折中方案: LSM6DSO32**

- 性能介于两者之间
- 价格适中（¥6-10）
- 功耗表现良好

### 2.3 传输方案对比

| 方案 | BLE模组 | WiFi模组 | BLE+边缘网关 |
|------|---------|---------|-------------|
| **典型模组** | ESP32-C3, nRF52 | ESP8266, RTL8720 | ESP32 + 本地网关 |
| **传输速率** | ≤2Mbps | ≤150Mbps | 按需 |
| **延迟** | 20-50ms | 5-20ms | 边缘预处理后低延迟上报 |
| **功耗** | 低 | 中 | 中（边缘网关供电） |
| **适用场景** | 单设备独立 | 健身房固定场景 | 多设备集群 |
| **与现有BLE适配器兼容性** | 高 | 中 | 高 |

**推荐: BLE模组 (ESP32-C3)**

理由：
- 现有BLE适配器框架无需重构，仅需添加新的GATT服务
- ESP32-C3支持BLE 5.0，功耗低，价格便宜（¥8-12）
- 社区生态成熟

## 3. 数据质量评估

### 3.1 IMU原始数据满足干预引擎输入要求分析

**干预引擎期望的活动数据类型** (参考 exercise_config.yaml):

```
- steps (计步)
- calories (卡路里消耗)
- activeMinutes (活动时间)
- distance (距离)
```

**IMU可提供的数据**:

| IMU数据 | 可提取的活动指标 | 是否满足干预引擎 |
|---------|----------------|-----------------|
| 加速度原始值 (ax, ay, az) | 运动强度、姿态变化 | 需预处理 ⚠️ |
| 角速度原始值 (gx, gy, gz) | 旋转动作、关节角度 | 需预处理 ⚠️ |
| 推算的次数/组数 | **reps, sets** (力量训练特有) | **可直接输入** ✅ |
| 运动类型识别 | 推/拉/蹲/举分类 | 需ML模型 ⚠️ |
| 运动轨迹 | 完成度评估 | 需边缘计算 ⚠️ |

**结论**: IMU原始数据**不能直接**作为干预引擎输入，需要经过边缘计算处理后提取特征指标。

### 3.2 力量训练动作识别可行性

| 动作类型 | 识别难度 | 技术方案 | 可行性 |
|---------|---------|---------|-------|
| 深蹲 (Squat) | 中等 | 加速度积分 + 阈值检测 | **可行** ✅ |
| 硬拉 (Deadlift) | 中等 | 陀螺仪Z轴检测 + 加速度峰值 | **可行** ✅ |
| 卧推 (Bench Press) | 较低 | 垂直方向加速度检测 | **可行** ✅ |
| 哑铃弯举 (Bicep Curl) | 较高 | 多轴融合 + 角度阈值 | **可行，需调优** ⚠️ |
| 腿举 (Leg Press) | 较低 | 单轴加速度检测 | **可行** ✅ |
| 龙门架训练 | 高 | 多IMU协同 + 姿态估计 | **复杂，暂不推荐** ❌ |

**结论**: 基础力量训练动作（深蹲、硬拉、卧推、腿举）识别**技术上可行**，识别准确率预期可达85-90%（需实际数据验证）。

### 3.3 运动量估算精度

| 指标 | 估算方法 | 预期精度 |
|------|---------|---------|
| **次数** | 加速度峰值检测 | ±1次，准确率 >95% |
| **组数** | 休息间隔检测 | ±1组，取决于休息阈值设置 |
| **运动轨迹** | 陀螺仪积分 | 误差较大，需校准 |
| **卡路里** | 基于体重+动作类型+次数估算 | ±20%（参考值） |
| **运动时长** | 基于实际运动时间统计 | 精度高 |

**结论**: **次数和组数**识别精度高，可作为干预引擎的可靠输入。卡路里估算精度有限，需用户体重数据辅助。

## 4. 技术架构适配

### 4.1 现有BLE适配器扩展分析

**现有架构**:
```
BLEAdapter (ble.adapter.ts)
├── 标准 GATT 服务 (心率、血压、体脂等)
├── 设备类型推断: inferDeviceTypeFromServices()
└── 数据解析: parseHeartRateData, parseBloodPressureData 等
```

**IMU设备需要**:

1. **新的设备类型**: `DeviceType` 枚举需添加 `imu_sensor`
2. **新的GATT服务**: 自定义IMU服务UUID
3. **新的数据解析**: 解析加速度+角速度原始数据
4. **新的数据模型**: `IMUSensorDataSchema`

**改造工作量评估**:

| 改造项 | 文件 | 工作量 |
|--------|------|--------|
| 添加设备类型 | device.model.ts | 小 (1处枚举) |
| 添加IMU数据Schema | device.model.ts | 小 (新schema) |
| 添加BLE服务UUID | ble.adapter.ts | 小 (常量定义) |
| 添加IMU解析函数 | ble.adapter.ts | 中 (解析逻辑) |
| 数据管道扩展 | data.pipeline.ts | 中 (MetricExtractionProcessor) |
| 边缘计算模块 | 新增模块 | 大 (需独立开发) |

**结论**: 现有BLE适配器**可直接扩展**支持IMU设备，无需重构。

### 4.2 数据管道改造

**现有数据流**:
```
Device → BLEAdapter → DataPipeline → MetricExtractionProcessor → Kafka → TimescaleDB
```

**IMU数据流**:
```
IMU传感器 → BLEAdapter(IMU) → 数据验证 → 边缘计算(特征提取) → Kafka → 干预引擎
                                         ↓
                                   力量训练指标:
                                   - exercise_type
                                   - reps_count
                                   - sets_count
                                   - rest_duration
```

**边缘计算需求分析**:

| 数据量 | 采样率4kHz，6轴数据，每轴2字节 | ≈ 48KB/s |
|--------|-------------------------------|---------|
| 压缩后 (BLE传输) | 差分编码 + 压缩 | ≈ 4-8KB/s |
| 边缘预处理 | 峰值检测、特征提取 | 实时计算可行 |
| 上报频率 | 按组次计，次/组上报 | <100 bytes/次 |

**结论**: **边缘计算是必需的**，但计算量适中。ESP32-C3 (240MHz, 520KB SRAM) 足够承担。

### 4.3 推荐架构方案

```
┌─────────────────────────────────────────────────────────────┐
│                    传统力量器械 + IMU模组                    │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  ICM-42688 + ESP32-C3 BLE模组                        │   │
│  │  - 实时采集加速度/角速度                              │   │
│  │  - 本地特征提取 (峰值检测、动作识别)                   │   │
│  │  - 按组次上报 (次数、组数、休息时长)                   │   │
│  └──────────────────────┬────────────────────────────────┘   │
│                         │ BLE 5.0                            │
└─────────────────────────┼───────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                  现有BLE适配器 (扩展)                        │
│  - 新增 IMU GATT Service (自定义UUID)                       │
│  - 新增 IMUDataSchema                                       │
│  - 新增 imu.parseIMUData()                                  │
└─────────────────────────┬───────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                  数据管道 (扩展MetricExtractionProcessor)   │
│  - 新增 exercise_type, reps_count, sets_count 提取          │
│  - 新增 activity_intensity 计算                             │
└─────────────────────────┬───────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│                  干预引擎 (输入适配)                         │
│  - 现有 activity_tracker 类型可直接接收                       │
│  - 无需修改核心逻辑                                          │
└─────────────────────────────────────────────────────────────┘
```

### 4.4 关键风险与缓解

| 风险 | 级别 | 缓解措施 |
|------|------|---------|
| IMU安装位置影响数据质量 | 高 | 制定标准化安装规范，提供安装指南 |
| 动作识别准确率不足 | 中 | 分阶段交付，先实现次数统计，再迭代识别模型 |
| BLE传输丢包 | 低 | 边缘计算本地缓存，连接恢复后补传 |
| 电池续航不足 | 低 | 选择低功耗芯片，关闭不必要的传感器 |

## 6. 实施建议

### 6.1 分阶段实施计划

**Phase 1: 技术验证 (2-3周)**
- 采购IMU模组样品 (ICM-42688 + ESP32-C3)
- 搭建测试环境，验证BLE数据传输
- 验证力量训练动作识别算法（次数/组数）
- 输出: 原型验证报告

**Phase 2: 边缘计算开发 (2-3周)**
- 开发ESP32固件，实现特征提取
- 开发BLE适配器IMU扩展
- 开发数据管道IMU处理器
- 输出: 可联调的模块

**Phase 3: 集成测试 (1-2周)**
- 与干预引擎联调
- 数据质量验证
- 边界情况测试

**Phase 4: 小批量试点 (2-4周)**
- 在客户B处安装10-20台设备
- 收集真实使用数据
- 迭代优化识别算法

### 6.2 技术储备要求

| 能力 | 当前状态 | 所需行动 |
|------|---------|---------|
| BLE开发 | 有框架，缺实现 | 引入noble库，完成BLE适配器实现 |
| 嵌入式C/C++ | 无 | 招募或外包ESP32固件开发 |
| IMU数据处理 | 无 | 内部算法团队或外包 |
| 力量训练动作识别 | 无 | 调研现有开源方案 (如ExerciseAI) |

### 6.3 关键技术决策点

1. **自研 vs 外购IMU模组**: 建议初期外购成熟BLE+IMU模组（如Seeed Studio的产品），加速验证
2. **动作识别自研 vs API**: 初期使用边缘端简单阈值法，验证可行性后再投入ML模型
3. **安装方式**: 磁吸 + 绑带，兼容不同器械

## 7. 结论

### 7.1 总体评估

| 维度 | 评估结果 | 说明 |
|------|---------|------|
| 技术可行性 | **可行** ✅ | 现有架构可扩展，IMU技术成熟 |
| 数据质量 | **基本满足** ⚠️ | 次数/组数精度高，卡路里估算需优化 |
| 工程复杂度 | **中等** | BLE适配器扩展工作量可控，边缘计算需独立开发 |
| 成本可控性 | **良好** | 单设备增量成本¥47-65，可通过溢价覆盖 |
| 商业价值 | **高** | 打开宁津传统设备存量市场 |

### 7.2 建议行动

1. **立即启动**: 采购IMU样品模组进行技术验证
2. **架构决策**: 采用ICM-42688 + ESP32-C3方案
3. **资源规划**: 投入嵌入式开发资源（可考虑外包）
4. **客户沟通**: 与客户B确认设备安装规范和试用计划

### 7.3 关键成功因素

- IMU传感器安装位置标准化
- 边缘计算算法的准确率和稳定性
- 与现有BLE基础设施的无缝集成
