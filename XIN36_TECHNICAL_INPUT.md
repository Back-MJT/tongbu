## 背景

客户B（传统力量器械厂商）希望通过在现有设备上加装 IMU 传感器实现数字化能力，数据通过 BLE 传入昕动数字底座。

本文档为 XIN-36 工程评估提供完整的产品需求和技术输入。

## 一、IMU 传感器选型评估

### 1.1 候选方案对比

| 方案 | 芯片 | 采样率 | 功耗 | BLE | 尺寸 | 成本 | 推荐 |
|------|------|--------|------|-----|------|------|------|
| MPU-6050 | Invensense | 最高 1kHz | 低 | ❌ 需外接 | 小 | ¥15-30 | PoC验证 |
| **MPU-6500** | Invensense | 32kHz | 低 | ❌ 需外接 | 小 | ¥30-60 | **试点首选** |
| BMI160 | Bosch | 1.6kHz | 极低 | ❌ 需外接 | 小 | ¥30-60 | 备选 |
| BMI088 | Bosch | 1.6kHz | 中 | ❌ 需外接 | 中 | ¥80-150 | 高端场景 |
| ICM-20948 | TDK | 9轴融合 | 中 | ✅ 内置BLE | 小 | ¥80-120 | 研发阶段 |
| BLE+MPU 组合 | NRF52840+MPU6500 | 100Hz+ | 低 | ✅ | 中 | ¥80-120 | 推荐方案 |

**推荐方案**: NRF52840 (BLE5.0 SoC) + MPU-6500 (6轴IMU) 组合
- 优势: BLE5.0 内置，功耗低，支持长距离，SDK成熟
- 劣势: 需要自己设计PCB (或使用模块)
- 替代: 直接采购 LilyGO T-Beam (¥50-80，带ESP32+BLE+IMU)

### 1.2 关键参数指标

| 参数 | 要求值 | 说明 |
|------|--------|------|
| 加速度测量范围 | ±16g (最低), 推荐 ±32g | 力量训练峰值加速度可能很大 |
| 陀螺仪测量范围 | ±2000°/s | 捕捉快速旋转动作 |
| 采样率 | **≥ 100Hz** | 动作分析需要足够时间分辨率 |
| ADC分辨率 | ≥ 16bit | 保证数据精度 |
| BLE距离 | ≥ 10m | 健身房场景需求 |
| 电池续航 | **≥ 8 小时连续采集** | 一次训练约1-2小时 |
| 防水等级 | IP67 | 运动出汗场景 |

## 二、IMU 数据传输方案

### 2.1 BLE 传输方案

**BLE GATT Profile 设计**:

| Characteristic | UUID | 属性 | 说明 |
|---------------|------|------|------|
| IMU Data | `0x1101` | NOTIFY | 加速度+角速度原始数据 (20 bytes/帧 @ 100Hz) |
| Device Info | `0x1102` | READ | 传感器ID、固件版本、电量 |
| Config | `0x1103` | WRITE | 采样率配置 (50/100/200Hz) |
| Command | `0x1104` | WRITE | 开始采集/停止采集/校准命令 |

**BLE 数据帧格式** (每帧 20 bytes, BLE MTU限制):
```
| Byte 0 | Byte 1-2 | Byte 3-4 | Byte 5-6 | Byte 7-8 | Byte 9-10 | Byte 11-12 | Byte 13-14 |
| 帧头   | accel_x  | accel_y  | accel_z  | gyro_x   | gyro_y    | gyro_z     | timestamp  |
| 0xAA   | int16    | int16    | int16    | int16    | int16     | int16      | uint16     |
```

> 注: 100Hz采样时，每秒产生50帧 BLE通知，实际传输速率约 1KB/s，在BLE4.2限速下可接受

### 2.2 边缘网关设计

由于 BLE 连接距离有限 (≤10m)，建议在健身房部署边缘网关 (树莓派/NUC):

```
IMU传感器 (BLE)
    │ ≤10m
    ▼
边缘网关 (Raspberry Pi 4 / NUC)
    │ 有线/WiFi
    ▼
昕动云端 (Kafka)
```

**边缘网关职责**:
1. BLE 扫描和连接多个 IMU 传感器
2. 数据缓存 (网络中断时本地缓存 ≤24h)
3. 协议转换 (BLE → MQTT/HTTP → 云端)
4. 基础数据预处理 (滤波、压缩)

## 三、运动数据转换算法需求

### 3.1 算法 pipeline

```
IMU原始数据 (100Hz)
    │
    ▼
┌─────────────┐
│  数据预处理   │  ← Kalman滤波/滑动均值去除噪声
└─────────────┘
    │
    ▼
┌─────────────┐
│  姿态估计    │  ← 四元数/欧拉角，Madgwick或Mahony算法
└─────────────┘
    │
    ▼
┌─────────────┐
│  动作分段    │  ← 检测动作开始/结束点，识别每次重复
└─────────────┘
    │
    ▼
┌─────────────┐
│  动作分类    │  ← 基于时序特征的分类模型 (DTW / 1D-CNN)
└─────────────┘
    │
    ▼
┌─────────────┐
│  强度评估    │  ← 基于速度和加速度峰值估算运动强度
└─────────────┘
    │
    ▼
HealthMetrics (统一Schema)
```

### 3.2 各算法模块详细需求

#### 3.2.1 姿态估计 (Attitude Estimation)
- **算法**: Madgwick AHRS (开源, 计算量小) 或 Mahony AHRS
- **输入**: 6轴 IMU (accel + gyro) @ 100Hz
- **输出**: 四元数 (qx, qy, qz, qw) 或欧拉角 (roll, pitch, yaw)
- **精度要求**: 动态误差 < 5° (相比光学动作捕捉系统)
- **计算平台**: 边缘网关 (Python/C++) 或云端 (取决于延迟要求)

#### 3.2.2 动作分段 (Action Segmentation)
- **目标**: 将连续动作流切分为独立的"一次重复"
- **方法**: 
  - 峰值检测法 (加速度方差/信号能量峰值)
  - 零速度检测法 (Zero Velocity Update - ZUPT)
- **输出**: Array of `{ startTime, endTime, peakAccel }`
- **难点**: 不同动作模式差异大 (卧推 vs 深蹲)，需分类后分段

#### 3.2.3 动作分类 (Exercise Classification)
- **目标**: 识别力量训练动作类型 (卧推/深蹲/硬拉/划船等)
- **方法**: 
  - 1D-CNN 时序分类模型
  - 输入特征: 加速度三轴 + 角速度三轴 + 四元数 (窗口 2秒)
- **训练数据需求**: 每个动作类型 ≥ 100 样本 (用户穿戴传感器录制)
- **初期方案**: 先做2-3个高频动作的分类 (卧推、深蹲)，剩余标为"其他"
- **准确率目标**: > 85% (初期), > 95% (迭代后)

#### 3.2.4 运动强度评估
- **指标**: 
  - 峰值加速度 (g)
  - 动作速度 (基于积分)
  - 估算 RM (Repetition Maximum)
- **RM 估算**: 基于负荷-速度关系 (Jidov SE)
  - `1RM = load / (1.0278 - 0.0278 × reps)` (Epley公式)
  - 对于 IMU 方案: 需要用户输入负荷，或通过速度和历史数据估算

## 四、安装方案评估

### 4.1 传感器安装方案对比

| 方案 | 安装位置 | 覆盖动作 | 实施难度 | 数据可靠性 | 推荐阶段 |
|------|---------|---------|---------|-----------|---------|
| 器械固定 | 固定在训练器械上 | 坐姿推胸、腿举等固定轨迹 | 低 | 中 | 试点首选 |
| 手腕绑缚 | 用户手腕/前臂 | 哑铃、杠铃类自由动作 | 中 | 高 | 试点次选 |
| 腰部绑缚 | 用户髋部/腰带 | 深蹲、硬拉等髋部主导 | 中 | 高 | 扩展 |
| 多传感器组合 | 器械+手腕 | 所有主要力量训练 | 高 | 最高 | 量产阶段 |

**试点推荐**: 
- 初期: 器械固定方案 (5-10台设备)，快速验证
- 同步: 手腕传感器方案 (10-20个用户)，验证算法精度

### 4.2 夹具设计要求

- **重量**: < 50g (不影响训练)
- **固定方式**: 弹性绑带/魔术贴，不破坏设备
- **角度**: 传感器 Y轴需与器械运动方向对齐 (或通过算法校准补偿)
- **防水**: IP67 (运动出汗)
- **成本**: 单件 < ¥30 (批量)

## 五、数据质量标准

| 指标 | 要求 | 测试方法 |
|------|------|---------|
| 采样率稳定性 | 100Hz ± 5% | 时间戳间隔分析 |
| 数据完整率 | ≥ 95% (每组训练) | BLE连接日志统计丢包率 |
| 传感器漂移 | < 2°/min (静止时) | 静止10分钟标定测试 |
| BLE连接成功率 | ≥ 99% | 长时连接测试 |
| 端到端延迟 | < 500ms | 网络延时+处理延时测试 |
| 电池续航 | ≥ 8h 连续采集 | 实际充放电测试 |

## 六、技术风险与缓解措施

| 风险 | 等级 | 缓解措施 |
|------|------|---------|
| IMU 传感器漂移导致动作识别不准 | 🟡 中 | 每次训练前自动校准(静止3秒)；算法端做漂移补偿 |
| BLE 连接不稳定 (多设备干扰) | 🟡 中 | 使用 BLE 频道分类；边缘网关部署多个 BLE Adapter |
| 力量训练动作多样，分类模型初期不准 | 🔴 高 | 初期限制动作类型；持续收集数据迭代模型 |
| 电池续航不足 | 🟢 低 | 低功耗设计；休眠模式 |
| 多传感器时间同步问题 | 🟡 中 | 使用边缘网关统一时钟 (NTP)；每帧带本地时间戳 |
