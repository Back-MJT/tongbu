## 1. 执行摘要

本文档定义**数字底座MVP**和**干预引擎v0.1**的产品规格。核心目标是：

- **数字底座MVP**: 实现多协议设备统一接入，打通"设备→云端→数据服务"链路，支持5款以上核心健康设备
- **干预引擎v0.1**: 在运动干预和睡眠干预两个场景完成闭环验证，建立个体健康画像和基线评估能力

**关键假设**: 团队已具备后端架构师、嵌入式工程师、数据科学家到位

## 2. 数字底座MVP核心功能

### 2.1 多协议设备接入网关

| 功能 | 优先级 | 描述 |
|------|--------|------|
| BLE蓝牙接入 | P0 | 支持蓝牙低功耗设备发现、连接、数据订阅 |
| WiFi设备接入 | P0 | 支持局域网TCP/UDP协议设备发现和接入 |
| 设备抽象层 | P0 | 统一设备数据模型，屏蔽协议差异 |
| 设备注册/注销 | P0 | 设备生命周期管理 |
| 设备状态监控 | P1 | 在线/离线/故障状态心跳检测 |

**MVP边界**: 仅支持BLE和WiFi，不支持Zigbee；不包含OTA升级

### 2.2 统一数据Schema V1.0

```typescript
// 设备原始数据
interface DeviceData {
  deviceId: string;           // 设备唯一标识
  deviceType: string;         // 设备类型(如 power_bike, sleep_monitor)
  timestamp: number;          // 采集时间戳(ms)
  data: Record<string, any>;  // 设备特定数据
}

// 标准健康指标
interface HealthMetrics {
  userId: string;
  timestamp: number;
  metrics: {
    heartRate?: number;
    bloodOxygen?: number;
    bloodPressure?: { systolic: number; diastolic: number };
    weight?: number;
    bodyFat?: number;
    steps?: number;
    sleepDuration?: number;
    sleepQuality?: number;
    exerciseDuration?: number;
    exerciseIntensity?: number;
    caloriesBurned?: number;
  };
}
```

**MVP边界**: 仅支持已接入的5款设备数据格式

### 2.3 实时数据管道

```
设备 → 协议适配器 → 消息队列(Kafka) → 数据处理(清洗/转换) → 时序DB(TimescaleDB)
```

| 组件 | 技术选型 | MVP职责 |
|------|----------|---------|
| 消息队列 | Apache Kafka | 设备数据缓冲与分发 |
| 时序数据库 | TimescaleDB | 健康指标存储与查询 |
| 数据处理 | Flink/自定义 | 数据清洗、格式转换 |
| API网关 | Kong | 统一API入口、鉴权、限流 |

**MVP边界**: 单节点部署，不支持跨区域容灾；数据保留30天

## 3. 干预引擎v0.1产品规格

### 3.1 个体健康画像

| 数据维度 | 数据来源 | 用途 |
|----------|----------|------|
| 生理指标 | 设备采集 | 身体健康状态评估 |
| 行为数据 | 设备+问卷 | 生活习惯分析 |
| 健康目标 | 用户输入 | 干预方向定义 |
| 历史干预效果 | 系统积累 | 方案优化参考 |

**健康画像模型**:
```json
{
  "userId": "xxx",
  "profile": {
    "age": 35,
    "gender": "male",
    "healthGoals": ["weight_loss", "sleep_improvement"],
    "baselineScores": {
      "cardiovascular": 70,
      "metabolic": 65,
      "sleep": 55,
      "musculoskeletal": 75
    },
    "riskFactors": ["sedentary", "high_stress"],
    "preferences": {
      "exerciseTime": "morning",
      "exerciseType": ["cycling", "swimming"]
    }
  }
}
```

### 3.2 基线评估模型

- **评估周期**: 用户首次使用后连续7天数据采集形成基线
- **评估维度**: 心血管、代谢、睡眠、骨骼肌肉、压力
- **评分算法**: 规则评分 + 简单统计模型
- **输出**: 基线健康评分(0-100) + 风险预警

**MVP边界**: 仅支持运动和睡眠两个场景的基线评估

### 3.3 干预方案生成引擎 (v0.1)

| 组件 | 技术方案 | 说明 |
|------|----------|------|
| 规则引擎 | Drools/Business Rules | 基于用户画像和健康目标的规则匹配 |
| 方案模板 | 预定义运动/睡眠处方模板 | 运动类型、强度、频率、时长 |
| ML辅助 | 简单协同过滤 | 基于相似用户的历史方案效果推荐 |

**运动干预方案示例**:
```json
{
  "prescriptionId": "EX-20260408-001",
  "userId": "xxx",
  "type": "exercise",
  "recommendations": [
    {
      "day": 1,
      "exercise": "cycling",
      "duration": 30,
      "intensity": "moderate",
      "targetHR": 120,
      "frequency": "daily"
    }
  ],
  "adjustmentRules": {
    "if": "restingHR_increase > 10",
    "then": "reduce_intensity_by_20%"
  }
}
```

**睡眠干预方案示例**:
```json
{
  "prescriptionId": "SL-20260408-001",
  "userId": "xxx",
  "type": "sleep",
  "recommendations": {
    "bedtime": "22:30",
    "wakeTime": "06:30",
    "sleepDuration": 8,
    "preSleepRoutine": [
      "21:00 停止使用电子设备",
      "21:30 轻度拉伸",
      "22:00 白噪音助眠"
    ],
    "environment": {
      "temperature": 20,
      "darkness": "complete",
      "noise": "below_30db"
    }
  }
}
```

### 3.4 效果评估与反馈闭环

| 指标 | 测量方式 | 评估周期 |
|------|----------|----------|
| 依从性 | 方案执行记录 | 每日 |
| 生理指标变化 | 设备数据对比 | 每周 |
| 主观反馈 | 问卷调查 | 每两周 |
| 综合效果评分 | 指标加权计算 | 每月 |

**反馈流程**:
1. 每日采集用户执行数据
2. 每周对比基线指标变化
3. 根据效果调整方案参数
4. 每月生成健康报告

## 4. 首批验证场景产品方案

### 4.1 运动干预场景

**用户痛点**: 缺乏个性化运动指导，无法量化运动效果

**产品功能**:
- 运动处方生成(类型/强度/频率/时长)
- 实时运动数据采集(心率、时长、消耗卡路里)
- 运动效果追踪(运动前后指标对比)
- 方案动态调整

**MVP验收标准**:
- 用户完成运动处方后，心率、卡路里等数据正确采集
- 运动依从性数据可追踪
- 效果评估报告可生成

### 4.2 睡眠干预场景

**用户痛点**: 睡眠质量差，难以改善

**产品功能**:
- 睡眠监测(入睡时间、醒来次数、深睡/浅睡比例)
- 睡眠质量评分
- 个性化睡眠方案(作息时间、环境建议、睡前仪式)
- 效果追踪与方案优化

**MVP验收标准**:
- 睡眠数据正确采集(通过已接入设备)
- 睡眠质量评分可计算
- 干预方案可执行

## 5. 端到端数据链路

### 5.1 MVP数据流

```
┌─────────┐    ┌─────────────┐    ┌─────────┐    ┌─────────────┐
│ 设备层  │───→│ 设备网关    │───→│ 消息队列│───→│ 数据处理    │
│ (BLE/   │    │ (协议适配)  │    │ (Kafka) │    │ (实时清洗)  │
│  WiFi)  │    └─────────────┘    └─────────┘    └──────┬──────┘
└─────────┘                                              │
                                                         ↓
                                          ┌─────────────────────────┐
                                          │ 数据存储                │
                                          │ (TimescaleDB + Redis)   │
                                          └────────────┬────────────┘
                                                       │
                                                       ↓
┌─────────────┐    ┌─────────────┐    ┌──────────────┴────────────┐
│ 干预引擎    │←───│ 规则引擎    │←───│ API服务                     │
│ (效果评估)  │    │ (方案生成)  │    │ (用户画像/历史数据查询)     │
└─────────────┘    └─────────────┘    └───────────────────────────┘
                                                       │
                                                       ↓
                                          ┌───────────────────────────┐
                                          │ 管理后台 (MVP仪表盘)       │
                                          │ - 设备管理                 │
                                          │ - 用户管理                 │
                                          │ - 方案配置                 │
                                          │ - 效果看板                 │
                                          └───────────────────────────┘
```

### 5.2 核心API设计

| API | 方法 | 说明 |
|-----|------|------|
| /api/devices/register | POST | 设备注册 |
| /api/devices/{id}/data | POST | 设备数据上报 |
| /api/users/{id}/profile | GET/POST | 用户健康画像 |
| /api/users/{id}/baseline | POST | 发起基线评估 |
| /api/users/{id}/prescriptions | GET | 获取干预方案 |
| /api/users/{id}/prescriptions/{id}/execute | POST | 执行干预方案 |
| /api/users/{id}/reports/{period} | GET | 健康报告 |

## 6. 5款核心设备选型

基于宁津产业基础和MVP验证需求，建议选型如下：

| # | 设备类型 | 品牌/型号 | 协议 | 用途 | 优先级 |
|---|----------|-----------|------|------|--------|
| 1 | 功率车(动感单车) | 舒华/舒华X5 | BLE | 运动干预 | P0 |
| 2 | 普拉提床 | 伊梵/YP-501 | WiFi | 运动干预 | P0 |
| 3 | 智能手环/手表 | 小米手环9 | BLE | 生理监测 | P0 |
| 4 | 睡眠监测垫 | 菠萝斑马/非接触式 | BLE | 睡眠干预 | P0 |
| 5 | 智能体脂秤 | 有品/P1 | BLE | 身体指标 | P1 |

**选型理由**:
- **功率车+普拉提床**: 宁津本地有健身器材产业基础，已有样板接入经验
- **小米手环**: 生态成熟，数据格式标准化程度高
- **睡眠监测垫**: 非接触式，用户接受度高
- **智能体脂秤**: 家庭场景普及率高
