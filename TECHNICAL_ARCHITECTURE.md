## 1. 架构设计原则

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **渐进式架构** | Phase 1 采用模块化单体，降低复杂度；Phase 2 可平滑演进到微服务 |
| **简单胜于复杂** | 优先使用成熟开源方案，避免过度设计 |
| **YAGNI** | 只实现当前需要的功能，预留扩展点但不提前实现 |
| **数据优先** | 数据质量监控和可观测性前置设计 |

### 1.2 约束条件

- **预算限制**: Phase 1 优先使用开源方案，控制云服务成本
- **本地部署需求**: 考虑宁津本地化部署，可能需要边缘计算能力
- **安全合规**: 个人信息保护法、医疗数据规范需前置设计
- **扩展性**: 需支持 Phase 2/3 的扩展（10+ 机构、万级设备）

### 2.2 技术栈选型

| 层级 | 组件 | 技术选型 | 选型理由 |
|------|------|----------|----------|
| **接入层** | 设备网关 | Node.js + TypeScript | 高并发、事件驱动、成熟的 BLE/WiFi 库 |
| **消息队列** | Apache Kafka | 开源、成熟、社区活跃 | 适合大规模设备数据采集场景 |
| **数据处理** | Flink CDC | 实时数据同步、CDC | 增量同步、事件驱动 |
| **时序数据库** | TimescaleDB | PostgreSQL 扩展 | SQL 兼容、水平扩展、成熟稳定 |
| **关系数据库** | PostgreSQL | 开源关系型数据库 | 可靠性高、功能丰富 |
| **缓存** | Redis | 内存缓存 | 实时状态、会话、限流 |
| **API 网关** | Kong | 开源 API 网关 | 鉴权、限流、插件生态 |
| **后端服务** | Node.js/Express 或 Python/FastAPI | 待定 | 需要根据团队技能决定 |
| **规则引擎** | 自研简单规则引擎 | JavaScript/JSON | MVP 阶段规则简单，降低复杂度 |
| **前端** | React + Tailwind + Shadcn UI | 现代前端框架 | 快速开发、组件丰富 |
| **部署** | Docker + Docker Compose | 容器化 | Phase 1 简单部署、云原生平滑过渡 |
| **监控** | Prometheus + Grafana | 开源监控 | 成熟、集成度高 |

### 3.2 设备抽象模型

```typescript
// 统一设备数据模型
interface DeviceData {
  deviceId: string;           // 设备唯一标识 (MAC 地址或厂商 ID)
  deviceType: string;         // 设备类型 (power_bike, pilates_bed, xiaomi_band, sleep_mat, scale)
  deviceModel: string;        // 设备型号 (SH-X5, YP-501, MI-BAND-9 等)
  firmwareVersion: string;    // 固件版本
  timestamp: number;          // 采集时间戳 (毫秒)
  data: DeviceRawData;        // 设备原始数据
  metadata: {
    signalStrength?: number;  // 信号强度 (RSSI)
    battery?: number;         // 电池电量 (0-100)
    online: boolean;          // 在线状态
  };
}

// 设备原始数据 (每种设备类型不同)
type DeviceRawData = 
  | PowerBikeData 
  | PilatesBedData 
  | XiaomiBandData 
  | SleepMatData 
  | ScaleData;
```

## 4. 数据管道架构

### 4.1 实时数据流

```
设备 ──▶ 设备网关 ──▶ Kafka ──▶ Flink ──▶ TimescaleDB
                     │                │
                     │                │
                     ▼                ▼
                   Redis           PostgreSQL
                 (实时状态)       (业务数据)
```

### 4.2 Kafka 主题设计

| 主题名称 | 分区数 | 保留时间 | 用途 |
|----------|--------|----------|------|
| `device.raw` | 3 | 7 天 | 设备原始数据缓冲 |
| `device.parsed` | 6 | 30 天 | 解析后的标准数据 |
| `device.events` | 3 | 90 天 | 设备事件 (上线/下线/异常) |
| `user.health` | 6 | 1 年 | 用户健康指标 (敏感，需加密) |
| `intervention.log` | 3 | 1 年 | 干预方案执行日志 |

### 4.3 数据质量监控

| 监控指标 | 告警阈值 | 处理措施 |
|----------|----------|----------|
| 数据缺失率 | > 10% | 告警 + 记录 |
| 数据延迟 | > 5 秒 | 告警 + 排查 |
| 数据漂移 | 超过历史均值 3σ | 标记异常 |
| 设备离线率 | > 20% | 告警 + 通知用户 |

## 5. API 层设计

### 5.1 RESTful API 规范

**基础 URL**: `https://api.xindong.com/v1` (生产环境)

**认证方式**: Bearer Token (JWT)

**通用响应格式**:
```json
{
  "code": 0,
  "message": "success",
  "data": { ... },
  "timestamp": 1234567890
}
```

### 5.2 核心 API 列表

#### 设备管理
| API | 方法 | 说明 |
|-----|------|------|
| `/api/v1/devices` | POST | 注册新设备 |
| `/api/v1/devices/:id` | GET | 获取设备信息 |
| `/api/v1/devices/:id` | DELETE | 注销设备 |
| `/api/v1/devices/:id/data` | POST | 设备数据上报 |
| `/api/v1/devices/:id/status` | GET | 设备状态 |

#### 用户管理
| API | 方法 | 说明 |
|-----|------|------|
| `/api/v1/users` | POST | 创建用户 |
| `/api/v1/users/:id` | GET | 获取用户信息 |
| `/api/v1/users/:id/profile` | GET/POST | 用户健康画像 |

#### 干预引擎
| API | 方法 | 说明 |
|-----|------|------|
| `/api/v1/users/:id/baseline` | POST | 发起基线评估 |
| `/api/v1/users/:id/baseline` | GET | 获取基线评估结果 |
| `/api/v1/users/:id/prescriptions` | GET | 获取干预方案列表 |
| `/api/v1/users/:id/prescriptions` | POST | 生成新方案 |
| `/api/v1/users/:id/prescriptions/:pid/execute` | POST | 执行方案 |
| `/api/v1/users/:id/effects` | GET | 效果评估结果 |

#### 数据查询
| API | 方法 | 说明 |
|-----|------|------|
| `/api/v1/users/:id/health-data` | GET | 健康数据查询 (支持时间范围) |
| `/api/v1/users/:id/reports/:period` | GET | 健康报告 (daily/weekly/monthly) |

## 8. 安全与合规

### 8.1 数据安全

| 措施 | 说明 |
|------|------|
| **传输加密** | 全链路 HTTPS/TLS |
| **存储加密** | 数据库字段级加密 (敏感健康数据) |
| **访问控制** | RBAC 角色权限 + API 级别鉴权 |
| **数据脱敏** | 导出/报告时自动脱敏 |
| **审计日志** | 核心操作记录审计日志 |

### 8.2 合规设计

| 合规项 | 设计措施 |
|--------|----------|
| **个人信息保护法** | 用户同意授权、数据最小化、删除权支持 |
| **医疗数据规范** | 数据分级 (普通/敏感/高度敏感)、访问审批 |
| **数据跨境** | 初始部署在境内，不涉及跨境 |
| **等保三级** | Phase 2 考虑 (当前 Phase 1 非必需) |

### 8.3 数据分级

| 级别 | 示例 | 存储要求 | 访问要求 |
|------|------|----------|----------|
| L1 普通 | 运动时长、设备型号 | 可明文 | 登录用户 |
| L2 敏感 | 心率、体重、睡眠 | 加密存储 | 授权 + 审计 |
| L3 高度敏感 | 病历、基因数据 | 加密 + 脱敏 | 审批 + 双录 |
